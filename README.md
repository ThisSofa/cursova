# Система обліку майна установи

## Призначення
Веб-застосунок для обліку майна установи з функціями:
- create: додавання майна;
- get: перегляд і пошук;
- update: редагування записів;
- delete: видалення записів;
- bulk selection: масовий вибір записів;
- bulk update/delete: масові операції над вибраними записами;
- сортування за категоріями та контроль стану майна.

## Чи достатньо однієї таблиці
Для поточного ТЗ достатньо однієї таблиці items.

Переваги однієї таблиці:
- простіша реалізація і захист курсової;
- швидкий запуск без складних зв'язків;
- повністю покриває сценарії CRUD, пошуку, сортування, контролю стану.

Коли варто розбивати на кілька таблиць:
- якщо з'являються довідники категорій, користувачі/ролі, історія руху майна;
- якщо потрібна аудит-історія змін або складна аналітика.

## Схема БД
Поточна схема (MySQL): одна таблиця items.

```sql
CREATE TABLE items (
    id INT AUTO_INCREMENT PRIMARY KEY,
    name VARCHAR(255) NOT NULL,
    category VARCHAR(255) NOT NULL,
    status VARCHAR(100) NOT NULL,
    quantity INT NOT NULL DEFAULT 1,
    description TEXT NULL,
    INDEX idx_items_name (name),
    INDEX idx_items_category (category)
);
```

## Архітектура
Проєкт запускається у Docker Compose та складається з 3 сервісів:
- db: MySQL 8.0, зберігає записи майна;
- api: Python FastAPI + SQLAlchemy, надає REST API;
- web: PHP + Apache, інтерфейс користувача.

Потік роботи:
1. Користувач відкриває UI на порту 8080.
2. PHP-сторінка звертається до Python API.
3. Python API читає/змінює дані у MySQL.
4. UI показує оновлені дані в таблиці.

## Структура проєкту
- docker-compose.yml: опис контейнерів та мережі.
- .env: локальні змінні середовища для контейнерів (порти, доступ до БД, URL API).
- .env.example: шаблон змінних середовища.
- backend/main.py: модель, підключення до БД, API-ендпоїнти.
- backend/requirements.txt: залежності Python.
- backend/Dockerfile: збірка контейнера API.
- frontend/index.php: вкладки, форми, таблиця, модальні вікна.
- frontend/Dockerfile: збірка контейнера фронтенду.

## REST API
- GET /items/: список майна з фільтрами search/category і сортуванням за category.
- GET /items/{item_id}: отримання одного запису.
- POST /items/: створення запису.
- PUT /items/{item_id}: редагування запису.
- DELETE /items/{item_id}: видалення запису.
- POST /items/bulk/delete: масове видалення за списком ids.
- POST /items/bulk/update: масове оновлення category/status за списком ids.

## Логіка UI
Сторінка має 2 вкладки:

1. Створення запису:
- форма create (назва, категорія, стан, кількість, опис);
- після успішного створення показується повідомлення.

2. Список майна:
- таблиця всіх записів (get);
- пошук за назвою і фільтр за категорією;
- кнопка Редагувати відкриває поп-ап з формою update;
- кнопка Видалити відкриває поп-ап підтвердження delete;
- чекбокси для bulk selection;
- масове видалення через поп-ап підтвердження;
- масове редагування category/status через поп-ап форму.

## Запуск
Вимоги:
- Docker Desktop;
- ОС Windows 10/11 або інша ОС з Docker.

Кроки:
1. Перейти до каталогу проєкту.
2. Налаштувати значення у файлі .env (паролі БД, порти, URL).
3. Виконати: docker compose up --build -d
4. Відкрити у браузері: http://localhost:8080

## Перевірка сценаріїв
1. Створіть кілька записів у вкладці Створення запису.
2. Перейдіть у вкладку Список майна і перевірте get/search/filter.
3. Відкрийте поп-ап Редагувати і збережіть зміни.
4. Видаліть один запис через поп-ап підтвердження.
5. Виберіть кілька записів чекбоксами та виконайте bulk update.
6. Виберіть кілька записів та виконайте bulk delete.

## Відповідь про вибір PHP
PHP для цього завдання є нормальним вибором:
- простий серверний рендеринг;
- швидка розробка без складного фронтенд-стеку;
- легко пояснити на захисті курсової.

Коли варто перейти на інший фронтенд:
- якщо потрібен складний інтерактивний UI, реальні SPA-сценарії та багато клієнтської логіки.
